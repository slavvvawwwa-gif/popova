# DESIGN QUALITY DIRECTIVES

Дополняет `design-system/MASTER.md`. MASTER фиксирует **палитру** (раздел 1).
Этот документ фиксирует **анти-дефолтную дисциплину**: что запрещено, какие шаги
обязательны до кода и как себя проверять после.

---

## 1. Почему выходит «плоско»

- Дефолтные Tailwind-классы (`rounded-lg`, `shadow-md`, `p-4`, `gap-4`,
  `text-gray-500`, `transition` без указанного easing) создают типовой вид
  независимо от палитры.
- Дефолтная шкала отступов Tailwind (4/8/16/24/32px) — узнаваемо «шаблонная».
- Дефолтные размеры текста (`text-sm/base/lg/xl`) — стандартная типографская
  шкала любого Tailwind-проекта.
- Рекомендации скилла — советы, а не принуждение. Без явного запрета на дефолты
  модель возвращается к привычным «безопасным» паттернам.

---

## 2. Явный запретный список (вставлять в промпты буквально)

При генерации/правке UI **запрещено** использовать:

- `rounded-lg`, `rounded-xl`, `rounded-2xl` как универсальное скругление —
  вместо этого: либо `rounded-none` (острые углы — строже), либо кастомное
  значение (например `rounded-[2px]` или `rounded-[24px]`, но обосновано
  композицией, не «по умолчанию»).
- `shadow-md`, `shadow-lg`, `shadow-xl` — дефолтные тени Tailwind. Если тень
  нужна — кастомная (`shadow-[0_1px_0_rgba(...)]` тонкая линия, не «мягкое
  облако»).
- Дефолтную шкалу spacing (`p-4`, `gap-4`, `space-y-6` и т.п. без обоснования) —
  расширьте собственной шкалой отступов (например на основе модульной сетки
  6/12/18/30/48/72/120px) и используйте её.
- `transition` / `transition-colors` без `duration` и `ease-*` — все переходы
  обязаны иметь явный `duration-[Xms]` и кастомную `ease-[cubic-bezier(...)]`,
  не `ease-in-out`.
- `text-gray-400/500/600` для вторичного текста — вторичный текст задаётся через
  `opacity` основного цвета текста (`text-primary/40` и т.п.), это даёт более
  цельную палитру.
- Стандартные размеры заголовков (`text-2xl`, `text-4xl`) — используйте `clamp()`
  с нестандартными значениями, привязанными к композиции, а не к готовой шкале.

---

## 3. Обязательные шаги перед генерацией кода

Проговорить **текстом, до кода**:

1. Запустить скилл с конкретным запросом и показать вывод
   (`python3 .claude/skills/ui-ux-pro-max/scripts/search.py "..." --domain effects`
   и т.п.) — не просто «учти скилл», а получить конкретные числа/паттерны.
2. Сформулировать 2-3 конкретных «дизайн-решения» для компонента (например:
   «underline растёт через scaleX за 220ms cubic-bezier(0.65,0,0.35,1)»,
   «карточка имеет негативный margin -24px и перекрывает соседнюю на 1/8 ширины») —
   конкретику, а не общие слова «сделаю красиво и асимметрично».
3. Указать конкретные значения spacing/timing/easing/aspect-ratio для ЭТОГО
   компонента — не «стандартные», а выведенные из композиции страницы.

---

## 4. Самопроверка после генерации (обязательный шаг)

После генерации компонента/страницы — пройти по чек-листу и переписать то, что не
проходит:

- [ ] Есть хотя бы один элемент, который выходит за пределы стандартной сетки
      (overlap, negative margin, асимметричный offset)?
- [ ] Все `transition`/`animation` имеют кастомный `duration` + `cubic-bezier`,
      не дефолтный `ease`?
- [ ] Использована ли хоть одна нестандартная деталь — текстура/grain,
      mix-blend-mode, нестандартный курсор, декоративный крупный элемент
      (число, линия, типографский акцент)?
- [ ] Если убрать цвет (`--accent`) — композиция всё ещё интересна за счёт
      типографики/расположения, а не «пустая коробка с текстом»?
- [ ] Не повторяется ли визуально один и тот же «паттерн карточки» 3+ раза
      подряд без вариаций (размер/aspect-ratio/расположение)?

Если хотя бы один пункт не выполнен — переделать, **не добавляя новый компонент
сверху**.

## 6. Применение в ЭТОМ проекте (важно — стек отличается)

Этот проект — **Tailwind v4 (CSS-first)** и преимущественно **inline-стили +
CSS-токены**, а НЕ utility-классы. Поэтому запреты раздела 2 переводятся так:

| Директива (Tailwind-формулировка) | Как enforce-ить здесь |
|---|---|
| Не `tailwind.config` со шкалой spacing | Шкала живёт в `:root`/`@theme` в `src/app/globals.css` (`--space-*`, уже заведено). Использовать `var(--space-N)`, не «голые» `rem`-литералы. |
| Не дефолтная type-шкала `text-2xl/4xl` | `--text-*` и `--display-1/2` в `globals.css` (`clamp()` с композиционными значениями). Заголовки — класс `.display`/`.h-serif`. |
| Вторичный текст не `text-gray-*` | Через альфу основного цвета: `--text-body` (0.66, читаемый) и `--text-secondary` (0.4, только мета). Уже внедрено. |
| `transition` обязана иметь `duration` + кастомный `cubic-bezier` | В inline-стилях писать полностью: `transition: "transform 220ms cubic-bezier(0.65,0,0.35,1)"`. **Запрещён** `ease`/`ease-in-out` по умолчанию. Завести общие кривые как токены (`--ease-curtain` и т.п.). |
| Радиусы | По умолчанию острые углы (`0`). Скругление — только осознанное кастомное значение. |
| Тени | Только тонкие функциональные (`0 1px 0 rgba(...)`), не «облака». |

**«Дефолт-риск» именно здесь** — это россыпь инлайновых `rem`-литералов и прямой
`var(--text-secondary)` на читаемом тексте. Миграция на токены уже начата
(SectionHead, `.body`, `.eyebrow`, `.measure`, `--space-*`); новые/правленые
компоненты обязаны брать значения из токенов, а не вводить новые «магические» числа.

### Кривые easing — общий набор (завести в globals.css при первом использовании)

```
--ease-out-expo:  cubic-bezier(0.16, 1, 0.3, 1);     /* появление, reveal */
--ease-in-out-q:  cubic-bezier(0.65, 0, 0.35, 1);    /* underline, hover */
--ease-out-soft:  cubic-bezier(0.33, 1, 0.68, 1);    /* мелкие state-смены */
```

Длительности: микро-взаимодействия 160–240ms, появление секций 600–800ms,
lightbox/overlay 200ms. `prefers-reduced-motion` гасит всё (уже реализовано).
